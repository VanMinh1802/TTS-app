Proposed Approaches — Library Redesign

Choose the architectural approach for implementation

A

Incremental Enhancement

Keep existing codebase structure, enhance library page component incrementally

  • Add view toggle, filters, sort to current page.tsx
  • Add useCloudLibrary hook alongside useLocalLibrary
  • Minimal refactoring, reuse existing patterns
  • Faster to implement, but file may grow large

Pros

  • Fastest to ship
  • Low risk
  • Familiar patterns

Cons

  • File bloat risk
  • Harder to maintain
  • Tangled concerns
B

Modular Refactor

Break library into clean feature modules with proper component separation

  • Separate components: Toolbar, Grid, List, Card, SyncBar, FilterPanel
  • Custom hooks: useLibraryRecords (unified local+cloud), useLibrarySync
  • React Query for cloud API, IndexedDB abstraction layer for local
  • Unified data model bridging local ↔ cloud schemas

Pros

  • Clean architecture
  • Testable components
  • Easy to extend

Cons

  • More upfront work
  • Need migration path
C

Unified Sync Architecture

Treat local IndexedDB as cache of cloud data, online-first with offline fallback

  • Library page always fetches from cloud API (if pro) with IndexedDB as cache
  • Background sync worker for offline to online transitions
  • React Query with optimistic updates
  • Reuses lib/sync/ and lib/offline/ scaffold

Pros

  • Future-proof
  • Offline-first
  • Uses existing infra

Cons

  • Heavy upfront cost
  • Overkill for this scope
  • Sync conflict complexity
