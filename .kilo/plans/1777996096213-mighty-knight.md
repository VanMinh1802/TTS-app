# Homepage WebGL Light Mode Improvement

## Problem
Homepage WebGL (`#heroCanvas`) is at 9% opacity in light mode — essentially invisible against the white body background (`#FCFAFF`→`#FFFFFF`). The blue/purple shader ring effect that gives the homepage its visual identity disappears completely.

## Root Cause
`globals.css` line 187-188: `.light #heroCanvas { opacity: 0.09; }` was reduced too aggressively. On the white light mode background, a 9% opacity blue-purple shader is imperceptible.

## Changes

### 1. globals.css — Increase WebGL canvas opacity
```css
/* Before */
.light #heroCanvas {
  opacity: 0.09;
}
/* After */
.light #heroCanvas {
  opacity: 0.30;
}
```
Reason: 30% provides the blue/purple glow visible against white while staying subtle enough not to distract from text.

### 2. page.tsx — Add light-mode background layer behind WebGL
The WebGL canvas uses `gl.clearColor(0.0,0.0,0.0,0.0)` (transparent) and the body is white in light mode. Add a subtle purple-tinted backdrop div behind the canvas on the homepage to give the WebGL rings something to render against:
```jsx
// Add inside the homepage container, before <WebGLBackground />
<div className="absolute inset-0 pointer-events-none z-0 aether-bg-gradient opacity-40" />
```
This adds a `radial-gradient(circle at 50% 0%, rgba(99,102,241,0.05), transparent 70%)` behind the canvas.

### 3. Verify existing light-mode overrides for homepage text
Already handled in globals.css:
- `.light .text-[#F4F4F5]` → `#09090B` (line 201) ✓
- `.light .text-[#A1A1AA]` → `#3F3F46` (line 202) ✓  
- `.light h1.bg-gradient-to-r` → purple gradient (line 236) ✓
- `.light .bg-gradient-to-b.from-white` → purple gradient (line 238) ✓
- `.light .aether-badge` → purple styling (line 226) ✓
- `.light .aether-btn-primary` → adjusted shadow (line 353) ✓

## Audit Summary

Audited 13 pages + 12 components against the responsive spec (breakpoints: 320-767 mobile, 768-1023 tablet, 1024-1440 desktop, 1441+ large desktop; touch targets ≥44px; collapsing strategies).

## Phase 2: Layout & Breakpoint Fixes (Medium Priority)

### 2.1 Dashboard Stats Grid — Missing Tablet Breakpoint
- File: `dashboard/page.tsx` line 101
- Current: `grid-cols-1 md:grid-cols-3`
- Fix: `grid-cols-1 sm:grid-cols-2 md:grid-cols-3`
- Reason: Spec requires 2-col at tablet (768px+). Current jumps directly from 1→3.

### 2.2 Library List Table — Overflow on Mobile
- File: `LibraryList.tsx` + `LibraryCardRow.tsx`
- Fixed-width columns (`w-28`, `w-14`, `w-24`, `w-16`) total ~82px + gap — overflows at 320px
- Fix: On mobile (`< sm:`), hide the "Status" and "Actions" columns; reduce "Voice" column width. Make remaining columns flex-based. The "Actions" column is already `opacity-0 group-hover:opacity-100` which helps, but column reserve space is the issue.
- Alternative: Make the table horizontally scrollable with `overflow-x-auto` on the parent, which is simpler and preserves all data.

### 2.3 Studio Sidebar — Tablet Optimization
- File: `studio/page.tsx` line 247
- Current: `lg:grid-cols-[1fr_400px]` (sidebar only at ≥1024px)
- Fix: Add `md:grid-cols-[1fr_300px] lg:grid-cols-[1fr_400px]` for tablet sidebar at reduced width
- Note: On mobile (<768px), sidebar content stays below main content (current behavior is fine)
