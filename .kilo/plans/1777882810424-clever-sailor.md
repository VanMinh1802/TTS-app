# Deploy Frontend to Cloudflare Pages — Monorepo Setup

## Current State

- Root repo: detached HEAD with `frontend/` as gitlink (submodule reference)
- `backend/` — FastAPI, deploys separately
- `frontend/` — Next.js 16 with Turbopack, contains its own `.git` directory

## Goal

Single monorepo on GitHub. Frontend auto-deploys to Cloudflare Pages, backend deploys to Railway/Render.

## Plan

### Step 1: Convert frontend gitlink → regular directory

```bash
# Remove gitlink from parent repo
git rm --cached frontend

# Remove frontend's separate git history (keep files)
rm -rf frontend/.git

# Add all frontend files to root repo
git add frontend/
git commit -m "chore: convert frontend from gitlink to monorepo directory"
```

### Step 2: Add Cloudflare Pages adapter

```bash
cd frontend
npm install @cloudflare/next-on-pages
```

Update `next.config.ts`:

```typescript
import type { NextConfig } from "next";

const nextConfig: NextConfig = {
  output: "standalone",
  experimental: {
    optimizePackageImports: ["@phosphor-icons/react", "framer-motion"],
  },
};

export default nextConfig;

// Cloudflare Pages: use @cloudflare/next-on-pages
// Run `npx @cloudflare/next-on-pages` to build, or
// add to package.json: "build:cf": "npx @cloudflare/next-on-pages"
```

Wait — `output: "standalone"` is NOT compatible with Cloudflare Pages. Cloudflare uses the Edge Runtime which doesn't support Node.js `server.js`. Need to:

- Remove `output: "standalone"`
- Use `@cloudflare/next-on-pages` for the build step

Actually, looking at the Next.js 16 + Cloudflare compatibility:
- `@cloudflare/next-on-pages` generates Cloudflare Workers-compatible output
- Web Worker TTS (`tts-worker.ts`) should work in Edge Runtime since it uses Web APIs
- ONNX Runtime Web works in browser/worker context, not server

### Step 3: Update package.json build scripts

```json
{
  "scripts": {
    "dev": "next dev",
    "build": "next build",
    "build:cf": "npx @cloudflare/next-on-pages",
    "start": "next start"
  }
}
```

### Step 4: Remove Node.js dependencies from Edge Runtime

Cloudflare Pages runs on Workers (Edge Runtime). Need to check:
- `apiRequest` calls backend API at `http://localhost:8000` → change to production URL via env
- `lamejs` (MP3) → works in browser, not server
- `onnxruntime-web` → works in browser/worker
- `framer-motion` → works in browser

No server-side code runs — all pages are `"use client"`, API calls go to backend. Good.

### Step 5: Commit & push to GitHub

```bash
git add -A
git commit -m "chore: prepare monorepo for Cloudflare Pages deployment"
git remote add origin https://github.com/<user>/TTS-app-v2.git
git push -u origin main
```

### Step 6: Configure Cloudflare Pages

1. Go to https://dash.cloudflare.com → Workers & Pages
2. Create Application → Pages → Connect to Git
3. Select GitHub repo `TTS-app-v2`
4. Build settings:
   - **Build command:** `cd frontend && npx @cloudflare/next-on-pages`
   - **Build output directory:** `frontend/.vercel/output/static` (or wherever CF outputs)
   - **Root directory:** `frontend/`
5. Environment variables:
   - `NEXT_PUBLIC_API_URL` = `https://api.yourdomain.com` (backend URL)
6. Deploy

### Step 7: Backend deploy (separately)

```bash
# Railway.app or Render.com
cd backend
# Set root = backend/ in deploy settings
# Add env: DATABASE_URL, R2_*, JWT_SECRET_KEY, etc.
```

## ✅ Decision: Monorepo (1 Repo cho cả Backend + Frontend)

Rất tiện. Ưu điểm:
- 1 repo = single source of truth
- Cùng commit hash → dễ trace bug
- API changes sync tự nhiên
- Không cần submodule

**Deploy architecture:**
```
GitHub:  /TTS-app-v2
├── frontend/         → Cloudflare Pages (root dir: frontend/)
├── backend/          → Railway.app (root dir: backend/)
└── .gitignore
```

Cloudflare Pages và Railway đều hỗ trợ **chọn root directory** khi deploy từ monorepo.
